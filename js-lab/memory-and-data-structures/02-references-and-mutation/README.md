# 02. Reference & Mutation

Розуміння того, як JavaScript керує пам'яттю, є ключовим для вирішення найскладніших багів. Найчастіша причина несподіваної поведінки додатків — це мутації, коли зміна об'єкта в одному місці непомітно "ламає" код в іншому.

---

## I. Essential Частина

### 1. Примітиви: Копіювання за значенням

**Теза:** Примітивні типи (числа, рядки, булеві значення) завжди копіюються за значенням. Це означає, що кожна змінна отримує свою власну незалежну копію даних. 

**Приклад:**
```javascript
let salary = 5000;
let newSalary = salary; // створює фізичну копію числа 5000
newSalary = 6000;

console.log(salary);    // 5000 (не змінилося)
console.log(newSalary); // 6000
```

**Просте пояснення (Junior/Middle):**
Якщо ви запишете число в блокнот і дасте другу скопіювати його, він перепише це число у свій блокнот. Тепер у вас два різних записи. Якщо друг закреслить і змінить своє число, ваше залишиться цілим. Число `5` неможливо "мутувати" — його можна лише перезаписати повністю.

**Технічне пояснення (Senior):**
Двигун V8 зберігає примітиви безпосередньо у `Environment Record` всередині Execution Context (Stack). Коли виконується оператор `newSalary = salary`, двигун фізично аллокує новий блок у Stack і копіює туди 64-бітне значення (або записує як 31-бітний Small Integer - SMI). Ці блоки не мають спільного посилання.

**Edge Cases / Підводні камені:**
> [!WARNING]
> Рядки у JavaScript концептуально є примітивами і передаються за значенням. Але під капотом V8 може оптимізувати великі рядки через механізм String Interning (зберігати їх у Heap для пулу рядків). З точки зору розробника це не має значення — рядки **незмінні (immutable)**. Спроба змінити `str[0] = 'H'` нічого не зробить.

---

### 2. Об'єкти/Масиви: Shared Reference Values

**Теза:** Складні типи даних (Objects, Arrays, Functions) існують в єдиному екземплярі у Heap. Змінні зберігають та передають **reference value** на цей об'єкт. У JS це все одно передача **by value**.

**Приклад:**
```javascript
const user = { name: "Bob", role: "User" };
const admin = user; // копіює АДРЕСУ, а не сам об'єкт

admin.role = "Admin";

console.log(user.role); // "Admin" - неочікувано!
```

**Просте пояснення (Junior/Middle):**
Об'єкт схожий на великий будинок. Змінна `user` — це аркуш паперу, на якому записана адреса цього будинку (наприклад, Київ, Хрещатик 1). Короткий запис `const admin = user;` не будує другий будинок! Він просто створює копію аркуша з адресою. Оскільки тепер двоє людей мають ключі до одного будинку, якщо `admin` пофарбує двері, `user` теж побачить пофарбовані двері.

**Технічне пояснення (Senior):**
Алокація `user` відбувається у Heap Memory. У `Environment Record` записується лише посилальне значення (умовно `0x11B33`). Оператор привласнення копіює саме це значення. Тому правильніше казати не "pass by reference", а "pass by value of a reference". Це породжує **shared reference**, внаслідок чого мутація властивостей через будь-який ідентифікатор змінює один і той самий об'єкт у Heap.

**Edge Cases / Підводні камені:**
> [!CAUTION]
> Передача об'єкта у функцію і його зміна всередині — класичний антипатерн, який ламає логіку програми. `function boost(obj) { obj.power += 10; }` змінює оригінальний об'єкт. Функції мають бути чистими (Pure Functions) і повертати новий об'єкт.

> [!IMPORTANT]
> Коректна формула для JavaScript: **усе передається by value**. Для об'єктів значенням є посилання на об'єкт.

### 3. The Const Paradox

**Теза:** Ключове слово `const` забороняє змінювати посилання у Стеку, але воно абсолютно **не захищає** властивості об'єкта у Купі (Heap) від мутацій.

**Приклад:**
```javascript
const config = { theme: "dark" };

// Дозволено (Мутація Heap)
config.theme = "light"; 
config.newField = 42;    

// Помилка TypeError: Assignment to constant variable. (Спроба змінити Stack)
config = { theme: "blue" }; 
```

**Просте пояснення (Junior/Middle):**
`const` — це суперклей, яким ви приклеїли аркуш з адресою до столу. Ви більше не можете замінити аркуш на інший (`config = {...}`). Але `const` не стоїть вартовим біля самого будинку! Ви все ще можете поїхати за цією адресою і вибити вікно (`config.theme = "light"`).

**Технічне пояснення (Senior):**
Статичний аналізатор V8 та фаза компіляції маркують ідентифікатор `config` у Lexical Environment як такий, що не підлягає перезапису (Read-Only прив'язка). Проте цей захист працює виключно на рівні Environment Record. Доступ до пам'яті за вказівником для зміни властивостей (Property Access) не перевіряється контекстом `const`, тому що рушій делегує це політикам дескрипторів об'єкта (`writable`, `configurable`).

## II. Advanced Section (Deep Dive)

### 1. Shared Mutable State (Проблеми спільного стану)

Хоча JavaScript є гарантовано однопотоковим (Single-Threaded), концепція **Shared Mutable State** здатна створювати "race conditions" логічного рівня. 
Якщо у вас є глобальний об'єкт `AppState`, з яким працюють декілька незалежних асинхронних функцій або компонентів, непередбачувані мутації гарантовано зламають потік даних.
Саме тому сучасні архітектурні патерни (Redux, Zustand, React State) вимагають жорсткої **Іммутабельності (Immutability)** — тобто створення нових посилань при будь-якій зміні:
```javascript
const newState = { ...oldState, role: "Admin" }; // Нова адреса в Heap
```

### 2. Immutability Patterns (`Object.freeze()` / `Object.seal()`)

Щоб справді заморозити будинок і заборонити його змінювати, `const` недостатньо. Потрібно змінювати дескриптори самих об'єктів у Heap.

- `Object.freeze(obj)` — робить об'єкт повністю Read-Only (`writable: false`, `configurable: false`).
- `Object.seal(obj)` — забороняє додавати чи видаляти поля, але дозволяє змінювати існуючі.

> [!WARNING]
> **Shallow Freeze:** Ці методи не працюють рекурсивно! Якщо всередині замороженого об'єкта є масив або інший об'єкт, їх все одно можна мутувати. 
Щоб обійти це, потрібна глибока заморозка (Deep Freeze рекурсивно) або інструменти на кшталт бібліотеки `Immer`.

**Візуалізація (Advanced):**

> Ця візуалізація демонструє, як поводяться вкладені об'єкти (Nested Objects) під час `Shallow Copy` (Spread), `Deep Copy` (structuredClone) та `Shallow Freeze` (Object.freeze).

### 3. Performance of Copying

Коли ви хочете скопіювати об'єкт, щоб уникнути мутацій, ви стикаєтеся з різними підходами і варіантами продуктивності (Performance Allocation):

| Метод | Копія | Продуктивність | Опис |
| :--- | :---: | :---: | :--- |
| `Spread { ...obj }` | Shallow | ⚡ Дуже швидко | Копіює лише верхній рівень полів. Вкладені об'єкти все ще ділять посилання. |
| `Object.assign()` | Shallow | ⚡ Дуже швидко | Аналог spread, але трохи повільніший через виклик методу (Function Call Overhead). |
| `JSON.parse(JSON.stringify())` | Deep | 🐢 Повільно | Історичний хак. Втрачає функції, `Date`, `undefined`, `Map`, `Set`. |
| `structuredClone(obj)` | Deep | 🚀 Швидко / Native | Сучасний нативний стандарт. Добре працює для більшості складних структур даних, включно з циклічними посиланнями. |

Використовуйте `structuredClone`, коли вам треба гарантовано розірвати зв'язок між глибоко вкладеними посиланнями без написання власного deep-copy коду. Але це не "срібна куля":

> [!WARNING]
> `structuredClone` не підходить, якщо:
> - вам треба зберегти прототипи користувацьких класів;
> - об'єкт містить функції;
> - копіювання саме по собі дорожче, ніж контрольована локальна мутація;
> - вам важлива не копія, а структурне розділення стану через persistent update pattern.

### 4. When This Matters / When It Doesn't

- **Важливо:** state management, reducer-патерни, cache invalidation, робота з props, конкурентні асинхронні сценарії.
- **Менш важливо:** короткоживучі локальні об'єкти всередині невеликих функцій, де shared mutation контрольована і не виходить назовні.
