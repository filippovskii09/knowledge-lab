# 03. Execution Context Stack (Стек викликів)

Щоб написати навіть найпростіший скрипт, рушію V8 потрібен механізм керування потоком виконання (Control Flow). Він повинен точно знати, який рядок коду виконується просто зараз, які змінні на ньому доступні та куди треба повернутися після завершення функції. Цим механізмом абсолютного контролю є **Стек викликів (Call Stack)** та **Контексти виконання (Execution Contexts)**.

Кожен виклик функції — це не просто стрибок по рядках тексту. Це розгортання абсолютно нового, ізольованого "світу", який має власний життєвий цикл.

---

## I. Що таке Execution Context (EC)?

**Теза:** Execution Context (Контекст виконання) — це абстрактне середовище або "капсула", в межах якої виконується ваш JavaScript-код. 

### Аналогія зі знімальним майданчиком
Уявіть, що ваш код — це сценарій фільму. 
Саме по собі написання функції в коді — це просто запис слів на папері. Але коли ви **викликаєте** цю функцію (`myFunc()`), рушій V8 розгортає справжній "знімальний майданчик": він наймає акторів та будує декорації (локальні змінні), налаштовує камери та освітлення (`this`). Усе це разом і утворює Контекст. Коли сцена відзнята (функція завершилася), майданчик демонтують.

### З чого складається контекст
Як ми згадували в попередніх статтях, всередині цього "контейнера" за специфікацією ECMAScript існують три найважливіші компоненти:
1. **Lexical Environment** — динамічний словник для `let`, `const` та функцій (що перемикається під час входження у блоки `{...}`).
2. **Variable Environment** — статичний словник для сумісності з `var`.
3. **`this` Binding** — посилання на об'єкт, у контексті якого викликається функція.

---

## II. Типи контекстів

**Теза:** Існує всього три типи контекстів виконання, причому 99% часу розробники мають справу лише з першими двома.

1. **Global Execution Context (GEC):** Базовий контекст за замовчуванням. Створюється **один раз** при старті скрипта. Його завдання — створити глобальний об'єкт (наприклад, `window` у браузері або `global` у Node.js) та прив'язати до нього слово `this` на верхньому рівні. Тільки один глобальний контекст може існувати у вашому потоці.
2. **Function Execution Context (FEC):** Створюється щоразу, коли функція **викликається** (увага: не коли оголошується/пишеться, а саме коли ви ставите круглі дужки `()`). Кожна функція має власне середовище зі своїм `this` та своїм `Lexical Environment`.
3. **Eval Execution Context:** Створюється при виклику функції `eval()`. Враховуючи проблеми з безпекою та руйнування V8 оптимізацій, використання `eval()` у сучасному JavaScript вважається моветоном.

---

## III. Call Stack (LIFO)

**Теза:** Call Stack (Стек викликів) — це структура даних, яка записує та керує тим, ***який саме*** Execution Context зараз активний. Він працює за принципом **LIFO (Last-In, First-Out: Останній прийшов — першим пішов)**.

### Приклад
```javascript
function funcB() {
  console.log("Inside B");
}

function funcA() {
  console.log("Inside A");
  funcB();
}

console.log("Global start");
funcA();
```

### Просте пояснення (Junior/Middle)
Оскільки JavaScript **однопотоковий** (Single-threaded), він може одночасно виконувати лише один рядок коду. Уявіть стопку тарілок. Найпершою ви кладете глобальну тарілку (GEC). Коли викликається `funcA()`, ви кладете тарілку А поверх глобальної. Коли `funcA` викликає `funcB`, ви ставите тарілку В на самий верх. 
Комп'ютер "їсть" (виконує код) завжди **тільки з найвищої тарілки**. Коли код `funcB` закінчується, тарілку В знімають (Pop). Тепер найвища — тарілка А, і процесор продовжує виконувати її далі.

### Технічне пояснення (Senior)
V8 використовує апаратний стек для керування потоком (Control Flow).
1. При старті скрипта: 엔진 робить `PUSH` (додавання) Global Execution Context у наразі пустий стек.
2. Виклик `funcA()` змушує рушій поставити поточний Global Execution Context на паузу (Suspend state) та зробити `PUSH` нового Function Execution Context для `А`. Тепер `А` — активний контекст.
3. Виклик `funcB()` так само заморожує `А` і робить `PUSH` контексту для `В`.
4. Коли виконання доходить до неявної інструкції `return` в кінці `funcB`, стек робить операцію `POP` (знімає верхній вузол). Контекст знищується. У цей момент рушій дістає зі стека попередника (контекст `А`), "розморожує" його стан і поновлює виконання рівно з того байт-коду, де він зупинився.

## IV. Рекурсія та Stack Overflow

**Теза:** Стек викликів є фізичною ділянкою пам'яті (RAM), що виділяється операційною системою. Отже, він має суворий ліміт на розмір. Безконтрольне наповнення стека призведе до фатальної помилки.

### Приклад
```javascript
function inception() {
  inception(); // Функція безкінечно викликає сама себе
}

inception(); // Maximum call stack size exceeded
```

### Просте пояснення (Junior/Middle)
Якщо з функцій робити "матрьошку", забувши прописати умову виходу (base case), рушій буде без зупинки класти нові й нові контексти одну на одну (PUSH). Врешті-решт стопка "проб'є стелю" виділеної пам'яті комп'ютера. Щоб браузер не завис насмерть, JS-рушій аварійно завершує роботу, викидаючи помилку `Stack Overflow` ("Переповнення стека"). 

### Технічне пояснення (Senior)
У рушії V8 розмір Call Stack лімітований і становить приблизно від 10,000 до 100,000 викликів залежно від середовища (у Node.js цей ліміт можна збільшити прапорцем `--stack-size`). Варто пам'ятати, що обсяг стека "з'їдають" не лише самі фрейми (Contexts), а й примітивні значення всередині `Lexical Environment`. 

> [!TIP]  
> У деяких мовах існує оптимізація хвостової рекурсії (Tail Call Optimization - TCO), яка дозволяє рушію "перевикористовувати" поточний фрейм стеку замість створення нового, якщо рекурсивний виклик є останньою операцією перед `return`. В специфікації ES6 TCO було додано, але на практиці у V8 (Chrome/Node) він досі вимкнений через складність у дебагінгу. Safari (JavaScriptCore) наразі єдиний рушій, що повноцінно підтримує TCO.

## V. Життєвий цикл контексту (Підсумок)

Будь-який Execution Context проходить 3 жорсткі стадії:

1. **Фаза Створення (Creation Phase):**
   - Додавання в Call Stack (PUSH).
   - Прив'язка ключового слова `this`.
   - Створення `Lexical Environment` / `Variable Environment`.
   - Підняття (Hoisting) та заповнення Environment Record (`var` стає `undefined`, `let`/`const` потрапляють у TDZ).

2. **Фаза Виконання (Execution Phase):**
   - Робота рушія "рядок за рядком".
   - Зняття блокувань TDZ (наповнення змінних реальними значеннями).
   - Виконання логіки або вкладені виклики інших функцій (що спричинить паузу поточного контексту і створення нового поверх).

3. **Фаза Знищення (Destruction Phase):**
   - Досягнення `return` або кінця тіла `}`.
   - Видалення фрейму зі стека (POP).
   - На цьому етапі спрацьовує Garbage Collector: він збирає весь Lexical Environment, що належав цьому контексту, і очищує пам'ять.

### Edge Cases / Підводні камені
> [!WARNING]
> **Пам'ять може "втекти" (Closure Escape)!** 
> На третій фазі (Destruction) Execution Context гарантовано вичищається з LIFO-стека. АЛЕ сам по собі `Environment Record` (фізичні змінні, що були створені) може **зберегтися** на купі (Heap), якщо всередині цієї функції ви створили іншу функцію і повернули її назовні. Це і є феномен **Замикання (Closure)**. Стек функцію знищив, а її змінні "пережили" її, бо на них існує посилання в Heap через прихований слот `[[Environment]]`.
